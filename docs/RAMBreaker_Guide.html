Scope

RAMBreaker is a workflow and correlation layer over Volatility — built to make the common investigations fast and the output shareable and defensible. It takes you from a memory image to a full report in one command on the everyday cases (Windows, common Linux distributions, macOS), and it is precise about exactly what it recovered on every run. When a target is a brand-new or unusual kernel, RAMBreaker's diagnostics tell you precisely what's needed — it never hands you a silent, empty report.

01

What it is

You give RAMBreaker a memory image. It figures out the operating system, picks the right Volatility engine, recovers the kernel symbols, runs a full plugin battery in parallel, mines the strings for artifacts, rebuilds the process and network picture, correlates it all, and writes a single self-contained HTML report. Every number in that report traces back to a raw Volatility plugin whose JSON is saved alongside — the report is for reading the story, the JSON is for proving it.

01 · DRIVE

Auto-detect & extract

Detects OS + engine + profile, then runs 9–48 plugins across a thread pool with an adaptive RAM guard.

02 · RECOVER

Kernel symbols

For Linux/macOS: download the matching ISF, patch a near-match to the kernel banner, rebuild it from the BTF inside the image itself, or build one from the distro's debug package — verifying each against the dump.

03 · CORRELATE

One story per process

Stitches process ↔ network ↔ command line ↔ modules ↔ hashes into a single narrative, not six disconnected tables.

04 · MINE

Strings → artifacts

A single pass over the strings extracts IOCs, browser history, and chat-app traces with low false-positive rates.

05 · DIAGNOSE

Loud, never silent

Catches Volatility's silent failures, corroborates the process list, and writes a precise, scrubbed fingerprint of exactly what broke.

06 · REPORT

One offline report

A self-contained report.html: 13 tabs, global search, opens offline with nothing to install.

07

Loud, never silent

A forensics report that is quietly incomplete is worse than one that crashes — an empty Processes tab looks exactly like a clean machine. Three mechanisms make that impossible, and every run gets an unambiguous status.

MECHANISM 01

Silent-failure demotion

A plugin that exits 0 but returns nothing while logging a struct/symbol exception is reclassified as a failure — its diagnostic output is kept, not discarded.

MECHANISM 02

Resume sidecar

A failed plugin leaves a marker so a re-run re-executes it instead of trusting stale or partial output.

MECHANISM 03

Process corroboration

The linked-list process view is cross-checked against a pool scan — "scan finds processes but the list is empty" is caught structurally.

Healthy

Corroborated & complete

Process sources agree, key tabs populated. Any failures are documented non-bugs. No fingerprint is written — nothing broke.

Degraded

Incomplete — verify

An empty key tab, or a symbol/struct issue. The report notes it and a scrubbed fingerprint records exactly which plugins and why.

Broken

Do not trust the tabs

No process source produced output, or the list is empty while a scan finds processes. Wrong symbols, a bad image, or hidden processes — flagged, never hidden.

08

Symbol acquisition, in order

Linux/macOS have no universal symbol server, so RAMBreaker climbs this ladder — first rung that validates against the dump wins.

0

Already installed

Do the currently installed symbols list real processes? If yes, persist the kernel info and go.

1

Exact community ISF

Scan every banner in the image, rank candidates, download each one's ISF, and verify it validates.

2

Banner-patched ISF

Patch a close match (e.g. a Debian ISF for a Kali kernel from the same source) to byte-match this banner.

3

Local DWARF build

Build an ISF from local kernel debug files with dwarf2json, then verify.

4

Built from the image itself

v6.2. Reconstruct the ISF from the BTF type graph + kallsyms the kernel carries in its own memory — no vmlinux, no package, no repo, no network. Rescues a kernel too new to be published, too old to still be hosted, custom-compiled, or analysed air-gapped. Needs CONFIG_DEBUG_INFO_BTF (~5.8+); falls through if absent. Deliberately ordered before the multi-GB download below.

5

Official debug package

Fetch the distro's -dbgsym/-dbg/debuginfo package over HTTPS (magic + SHA-256 verified), extract vmlinux, run dwarf2json, install, verify.

!

None validated → fail precisely

Record the parsed kernel banner and classify the failure, so the exact symbols can be refetched later — without ever needing the evidence image.

Engineered to be trusted

A forensics tool earns trust in the details, so the unglamorous parts are first-class.

Tested where it matters

A pure-logic suite pins the health, scrubbing, and report-encoding code; a "canary" drives the entire real pipeline against a fake Volatility — no memory image needed — so runtime regressions are caught in seconds. A pre-commit hook runs both before every commit. 166 pure-logic assertions plus a 21-assertion pipeline canary — 187 checks, all green.

Secure by construction

The report neutralizes hostile memory strings; every download is integrity-verified (HTTPS + magic + SHA-256); NTLM hashes are redacted in the report; and the project is structured so evidence never enters version control.

Evidence, not verdicts

Nothing is labeled "malware." Findings are observations — an unexpected parent, a process a scan sees but the list doesn't — so the analyst stays in control and the output stays defensible.

Reproducible & portable

The toolchain is pinned, the symbol builder is committed in-repo, and a run's symbol resolution is reproducible — so a finding can be reproduced from the same image and tag.

python3 crescent_toolkit.py # interactive menu python3 crescent_toolkit.py full -i image.raw -o ./results/ git config core.hooksPath .githooks # enable the test gate
